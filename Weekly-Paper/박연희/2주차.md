## Git에서 branch merge 방법들과 각 방법의 특징을 설명해 주세요.
- 현재 branch에서 다른 branch를 합칠 때 사용하는 것을 branch merge라고 합니다.
- branch merge에 세 가지 방법이 존재하는데
    
    1. Merge Commit / 2. Squash Merge / 3. Rebase Merge
    
1. **Merge Commit**
    - 일반적으로 많이 사용하는 merge 방법이며, 커밋 이력을 모두 남길 때 사용합니다.
    - 장점이자 단점은 모든 커밋과 분기했던 branch 히스토리가 남는다는 점입니다.
2. **Squash Merge**
    - 분기했던 branch에 있던 커밋을 모두 합쳐 하나의 새로운 커밋을 만듭니다.
    - 깔끔한 히스토리를 만드는 대신 중간 작업 커밋은 사라집니다. 그래서 잘못 사용하여 과도한 생략을 하게 되면 추후 변경 파악이 힘들 수 있습니다.
3. **Rebase Merge**
    - 분기했던 branch의 기준을 최신 Base로 설정하고, merge하는 방법입니다.
    - rebase를 하면 커밋들의 Base가 변경돼 커밋 해시 또한 변경 될 수 있습니다.
    - 머지 커밋을 남길 필요가 없는 merge의 경우 사용하면 좋습니다. 시간 순으로 깔끔하게 이어지는 대신 충돌 가능성이 높습니다.


## Git Flow 브랜치 전략에 대해 설명해 주세요.
**기능 개발**, **릴리즈 준비**, **버그 수정**, **배포** 등 소프트웨어 개발 라이프사이클을 각 브랜치에 명확하게 분리하여 관리하는 전략입니다.

**기준 브랜치 (항상 존재)**

- **`main`**
    - 실제 **배포**되는 안정된 코드로 **항상 배포 가능한 최종 버전 상태**로 유지되고 있습니다.
- **`develop`**
    - **기능 개발의 중심 브랜치**로 기능 추가, 버그 수정 등 개발이 이뤄집니다.
    - 릴리즈 준비가 끝나면 `main`으로 병합
    

**임시 브랜치 (필요할 때 생성 후 병합/삭제)**

- **`feature/*`**
    - 새로운 **기능 개발용 브랜치**로 `develop`에서 분기했다가 작업 후 다시 병합합니다.
- **`release/*`**
    - **배포 전 마무리 작업용 브랜치**로 테스트, 버그 수정하고 완료 후 `main`과 `develop`에 병합합니다. + 태그 기능을 써 관리를 한다.
- **`hotfix/*`**
    - **배포 중 발생한 긴급 버그 수정 브랜치**로 ****`main`에서 분기한 뒤 수정 후 `main`과 `develop`에 병합합니다.

- **장점**
    - 안정적인 배포 구조이며 복잡한 개발/배포 흐름 관리에 유리합니다.
    - 긴 개발 주기나 큰 프로젝트에 적합하다.
- **단점**
    - 브랜치가 많아지고 관리가 복잡해집니다.
    - 작은 프로젝트나 빠른 배포에는 과할 수 있음
